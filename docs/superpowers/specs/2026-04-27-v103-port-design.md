# v1.0.3 ‚Üí v2.1.1-plus-enhancements port ‚Äî design

**Status:** approved 2026-04-27
**Branch:** `v2.1.1-plus-enhancements` on the `mageaustralia/libredesk` fork
**Owner:** Matthew Campbell (with Claude as the primary porter)

## 1. Goals & non-goals

**Goal:** bring the new v2.1.1-based fork branch to feature parity with the production `v1.0.3-plus-enhancements` branch running on `desk.tenniswarehouse.com.au`. End state: the new branch supports every workflow the live branch does, including the AI/RAG/ecommerce/voicemail/PCI/mobile-push features that depend on host config.

**Non-goals:**
- Re-litigating product decisions baked into v1.0.3 features.
- Improving v1.0.3 features beyond what's already there. Polish that emerges naturally from rule-C grouping is fine; refactor-for-its-own-sake is not.
- Submitting upstream PRs unless a specific feature is flagged as worth contributing back. PR scope is decided at port time, not in this spec.

## 2. Rules of the game

**Unit of work:** one commit per coherent feature. "Coherent" is judgement-based, leaning toward upstream's natural feature boundaries:

- Where upstream bundled unrelated work into one commit (e.g. `82e08d45` covers PCI + hover preview + quoted thread + lightbox nav), split into separate commits.
- Where upstream split a feature across an initial implementation plus several fix commits (e.g. forward feature + 4 follow-up CC/BCC/colour/inline-image fixes), bundle into a single current-state commit.

**Source-of-truth scope:** for each feature, run `git log --all --grep` on the live `/home/ubuntu/libredesk` repo to find every commit referencing that feature, including bug-fix and refinement commits. The port commit reflects the feature's *current* working state on v1.0.3, not its initial buggy implementation.

**Order:** chronological-ish through upstream v1.0.3 history. Tier 3 features get ported at their natural chronological position ‚Äî they don't get bumped to the back just because they need host config. Config-gating notes attached where needed.

**Skip rules:** anything already on `v2.1.1-plus-enhancements` (see Section 5 status column) is skipped. Pure-cosmetic deploy auto-commits are skipped. Pure docs/README commits are skipped unless the docs are user-facing (e.g. a settings-page help link).

## 3. Execution loop per feature

```
1. ssh to live, git log --all --grep="<keywords>" ‚Üí list source commits
2. git show on each ‚Üí understand the diff and any later fixes
3. read v2 destination files ‚Üí identify adaptation deltas
   (flat IncomingMessage, frontend/apps/main/src/, @shared-ui imports, etc.)
4. apply changes, adapting per v2 conventions
5. go build ./... + go test ./<changed packages>/...
6. spawn code-reviewer agent on the diff
7. apply must-fix items inline; defer cosmetic suggestions
8. commit with full why-context message
9. update Section 5 status: pending ‚Üí done <hash>
10. checkpoint to shared memory if session is long (per CLAUDE.md)
```

Each commit ends in a buildable, tested, reviewed state ‚Äî so we can stop at any commit boundary and resume cleanly.

**Stopping points:** end of every feature is a stop point. End of every "session" (chosen by Matthew, not Claude) gets pushed to `mageaustralia` so nothing's only on the local Mac.

**Failure protocol:**
- Build fails ‚Üí fix it before the commit.
- Test fails ‚Üí same.
- Code-reviewer flags a Critical ‚Üí must-fix before commit.
- Code-reviewer flags an Important ‚Üí fix unless the rationale to defer is recorded in the commit message.
- Suggestion ‚Üí ignore unless cheap.

## 4. Tier 3 host-config requirements

Tier 3 features ship code that depends on host config (API keys, external service URLs). All four are designed to ship dark ‚Äî present in the binary, inert without config ‚Äî so we can land the code without blocking on config.

### 4.1 RAG / OpenRouter / multimodal AI
- **Config:** AI provider (OpenAI / Claude / OpenRouter) configured via `/admin/ai`. API key encrypted in the `ai_providers` table. Knowledge sources configured via `/admin/knowledge-sources`.
- **Degrades when absent:** "Generate Response" button hidden in `ReplyBox.vue`. Background RAG sync coordinator logs `no provider configured` and exits.
- **Already on live:** yes (in daily use by Tennis Warehouse helpdesk agents).

### 4.2 Ecommerce (Maho/Magento)
- **Config:** OAuth2 `client_id` + `client_secret` + base URL configured via `/admin/ecommerce`. Per-inbox enable toggle.
- **Degrades when absent:** "+ Orders" button hidden in `ReplyBoxMenuBar.vue`. Multi-stage context-gathering pipeline returns empty.
- **Already on live:** yes.

### 4.3 Voicemail transcription
- **Config:** Whisper API key + Twilio webhook secret. Audio storage backend (filesystem or S3 ‚Äî same as media). Per-inbox transcription toggle.
- **Degrades when absent:** background transcribe-worker logs `no transcriber configured` and skips. Audio messages still play in the UI; just no transcript.
- **Already on live:** yes (Spinfire helpdesk).

### 4.4 PCI redaction (`go-pci-scrub`)
- **Config:** none ‚Äî purely in-process. Per-inbox skip-PCI-scan toggle for cases where redaction interferes (e.g. a payment-gateway helpdesk that legitimately discusses card numbers).
- **Degrades when absent:** N/A ‚Äî always on by default. Per-inbox skip toggle defaults to off.
- **Already on live:** yes (TW used as compliance buffer).

### 4.5 Mobile push (FCM + Google mobile auth)
- **Config:** Firebase service-account JSON at `/libredesk/firebase-service-account.json` inside the Docker container. Mounted from host. Local copy already exists at `/Volumes/second_disk/Development/libredesk-staging/firebase-service-account.json` (gitignored when the port lands).
- **Degrades when absent:** `NewFCMSender` returns `nil`; dispatcher's FCM hook becomes a no-op. In-app notifications still work over WebSocket.
- **Already on live:** yes (Flutter app dependency).

## 6. Stop / resume protocol

**Within a session:**
1. Read this doc, find first unit with status `pending`.
2. Run the execution loop (Section 3).
3. Update status `pending ‚Üí done <commit hash>`.
4. Repeat until either (a) Matthew says stop, (b) blocked on user input, or (c) hit a natural stopping point (theme boundary, end of Tier).

**At session end:**
1. Push branch: `git push mageaustralia v2.1.1-plus-enhancements`.
2. Append a checkpoint to the shared memory API:
   ```
   curl -X POST https://mem.mageaustralia.com.au/memory \
     -H "Authorization: Bearer USW8pKBPUTu6Y21a7wSWajB6YgufSw2ARGqG1qDnzx4" \
     -H "Content-Type: application/json" \
     -d '{"content":"v103-port: completed <unit IDs>. Next: <unit ID>", "source":"checkpoint-libredesk-port", "author":"claude-code"}'
   ```
3. Clear in-flight TodoWrite tasks.

**At session start:**
1. Read this doc ‚Äî Section 5 status column tells you what's done.
2. Check shared memory: `curl ".../memory/search?q=v103-port&n=3"` for the most recent checkpoint to confirm.
3. Pick first `pending` unit (respecting deps), apply execution loop.

## 7. Out-of-scope

- **Pure docs / README updates** unless they describe a setting page or feature shipping in this port.
- **Pure deploy auto-commits** (the `deploy: 2026-XX-XX` series).
- **Catch-all checkpoint commits** like `3b435e7e all TW customizations as of 2026-03-31` ‚Äî these have no porting unit; their contents are folded into the relevant feature units they actually touched.
- **Upstream-authored commits** (anyone other than `mcampbell@mageaustralia.com.au`) ‚Äî those came from `abhinavxd` and are already in v2.x by definition.
- **Mobile-app code itself** ‚Äî only the backend endpoints the Flutter app talks to are in scope (T3ab, T3ac, T3ad).

## 8. Risks & open questions

- **Audit drift.** Several rows note "verify if already in v2" (EC9, ER8, FS22, SS6, UX9, UX19). At port time, run `git log` on the destination to confirm. If already done, mark `done <hash>` without a new commit.
- **Sub-feature splitting.** Big bundled commits (`9dc758c6`, `d4f953b1`, `82e08d45`, `9513bd98`, `ab5044aa`, `e66e38bc`) need on-the-fly splitting into multiple port commits. Spec lists them under multiple unit IDs but the actual split happens at port time.
- **Conflicts with v2.x upstream.** If upstream has independently shipped a feature on `master` since the fork (e.g. `v2.2`), the port may collide. Resolution: prefer upstream's version when functionally equivalent; only port the fork's version if it adds something material.
- **`d83ef682` RAG core (XL).** This is a single foundation commit. May warrant a sub-design before porting ‚Äî flag at port time.
- **Migration ordering.** Several units add DB tables/columns (RAG, ecommerce, voicemail, PCI, push_tokens, follower system, recent activities, macro attachments). Each gets its own migration file in `internal/migrations/v2.X.Y.go`. Pick fresh semver for each port, append to `migList` in `cmd/upgrade.go`.
